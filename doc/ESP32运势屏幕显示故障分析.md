# ESP32 太极运势界面文字显示错乱问题分析与修复文档

## 文档信息

- **设备**：ESP32 液晶 / 墨水屏显示设备
- **故障现象**：运势文本渲染残缺，仅显示「进，行。」「出行」，其余文字丢失

### 预期完整文案

```
今日运势
乾为天
今日宜进取，顺势而行。
宜：签约、出行
忌：熬夜、口舌
```

---

## 一、故障现象描述

- **图形资源正常**：太极阴阳图、外围各类功能图标（人物、定位、齿轮、爱心、日历等）均可正常渲染。
- **文本渲染异常**：仅后半段少量汉字正常显示，前文「今日运势、乾为天、签约、熬夜、口舌」完全消失。
- **文字排版错乱**：分行失效，仅中间太极区域存在残缺字符。

---

## 二、故障根因分层分析（按出现概率从高到低排序）

### 2.1 中文字库资源异常（最高概率）

**故障诱因**

- 字库文件裁剪不全：烧录的点阵 / GFX 中文字库未包含全部运势所需汉字，缺失「今、日、运、势、乾、签、约、熬、夜、口、舌」等字符，无对应点阵数据则屏幕空白。
- 文件系统存储空间不足：字库存放于 SPIFFS / LittleFS 分区，分区容量不足、上传中断导致字库文件损坏，加载时仅读取少量字符。
- 编码不匹配：代码字符串为 UTF-8 编码，字库仅支持 GB2312/GBK，字符编码映射失败无法渲染。

**对应修复方案**

- 重新生成完整字库，包含运势文案全部汉字。
- 增大 Flash 文件系统分区容量，重新上传完整 `font.bin` 字库文件。
- 统一编码标准：源码文件保存为 UTF-8，配套使用 UTF-8 映射点阵字库。

### 2.2 字符串缓冲区 / 堆内存溢出（ESP32 高频内存问题）

**故障诱因**

- 文本存储数组长度分配过小，完整运势字符串超出缓冲区长度，内存截断仅保留末尾字符。
- 堆内存资源耗尽：太极图形、外围图标提前占用大量内存，绘制长文本时内存分配失败，渲染中断。
- 字符串拼接函数 `sprintf` 未限制写入长度，缓冲区溢出篡改内存数据，文字残缺错乱。

**对应修复方案**

扩容文本缓冲区，预留充足存储空间：

```c
// 错误写法（容量不足）
char fortuneText[32];

// 修复写法
char fortuneText[128];
```

- 绘制文字前监控剩余堆内存，低于 2KB 时提前释放图片、图标缓存。
- 替换不安全的 `sprintf` 为长度限制安全函数 `snprintf`，防止溢出。

### 2.3 屏幕渲染绘制参数错误

**故障诱因**

- 文字绘制起始坐标越界：前文文字绘制 Y 坐标超出屏幕顶部可视范围，肉眼无法看见，仅底部「出行」落在太极可视区域。
- 文本自动换行逻辑失效，单行文字横向超出屏幕直接截断。
- 屏幕裁剪区域（clip）配置错误，仅中间太极区域允许绘制文字，顶部显示区域被屏蔽。

**对应修复方案**

分段调整文字绘制坐标，自上而下分层排版：

```c
drawString("今日运势",            屏幕居中X,  20, 字体);
drawString("乾为天",              屏幕居中X,  60, 字体);
drawString("今日宜进取，顺势而行。", 屏幕居中X, 100, 字体);
drawString("宜：签约、出行",       屏幕居中X, 140, 字体);
drawString("忌：熬夜、口舌",       屏幕居中X, 180, 字体);
```

- 移除错误的 `setClip()` 裁剪限制，或扩大裁剪区域覆盖全屏幕。
- 启用屏幕驱动自动换行接口，避免单行超长文字截断。

### 2.4 运势文本数据读取逻辑 Bug

**故障诱因**

- 存储运势文本的常量数组下标读取越界，仅截取字符串末尾片段。
- 从 SPIFFS 读取运势文本文件时，读取长度参数配置错误，仅读取末尾少量字符。
- 多任务并发冲突：屏幕刷新任务、文本读取任务并行执行，文本缓存被中途覆盖，渲染时数据丢失。

**对应修复方案**

串口打印校验原始文本数据，快速定位故障层：

```c
Serial.println(fortuneText);
```

- 串口输出残缺：数据读取 / 数组赋值逻辑存在问题。
- 串口输出完整、屏幕缺字：字库或屏幕绘制坐标问题。
- 读取文件时增加文件长度校验，补充读取失败重试逻辑。
- 屏幕刷新操作前增加互斥锁（Mutex），避免多线程篡改文本缓存。

### 2.5 硬件与显示驱动次要问题

- 屏幕刷新时序过快，显存未完整写入文字就执行刷新，出现文字残缺；可增加同步刷新逻辑或短暂延时。
- 文字配色对比度不足：深色背景 + 浅淡文字，视觉上呈现"看不见"的效果，调整文字前景色亮度。

---

## 三、分步调试定位流程（执行顺序不可颠倒）

### 3.1 串口数据源校验

将完整运势字符串通过串口打印，判断原始文本是否完整输出，区分「数据层故障」与「屏幕渲染层故障」。

### 3.2 单段文字独立绘制测试

单独绘制「今日运势」测试顶部文字是否显示，判断是否为坐标越界问题。

### 3.3 极简字库专项测试

制作仅包含前文汉字的最小字库，测试顶部文字是否渲染，验证字库缺失问题。

### 3.4 堆内存监控排查

文字绘制前打印剩余堆内存数值，确认是否存在内存不足导致渲染失败。

---

## 四、故障修复优先级总结

| 优先级 | 排查方向 | 关键动作 |
|--------|----------|----------|
| 一级 | 串口打印校验 | 校验原始文本完整性，区分故障层级 |
| 二级 | 渲染层排查 | 若文本完整 → 检查字库完整性 + 文字绘制坐标 |
| 三级 | 数据层排查 | 若文本残缺 → 扩容字符串缓冲区，修复文件 / 数组读取逻辑 |
| 四级 | 兜底排查 | 内存溢出、多线程冲突、屏幕裁剪驱动时序问题 |
